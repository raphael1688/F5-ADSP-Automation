# Deploy UC3 in Google Cloud

F5 BIG-IP with a self-signed front-door TLS cert, fronting a Docker-hosted comfy-capybara application. F5 API Security Local Edition (ASLE) runs on a dedicated Docker host and receives telemetry from BIG-IP via an iRule, an internal logging VS chain, and an external collector pool. Deployment is driven entirely by GitHub Actions; local execution is not supported.

This document has two parts:

- **Part 1 - Quickstart**: the minimum set of edits to get UC3 running, assuming you already have GCP + WIF + F5 entitlements wired up.
- **Part 2 - Detailed Reference**: architecture, IAM model, full configuration surface, verification, and troubleshooting.

---

# Part 1: Quickstart

## What gets built

```
F5 BIG-IP (single-NIC, PAYG)
  - Front-door cert: self-signed, generated by bigip-config
  - Management on 8443 locked to admin_src_addr
  - App VIP 443: pool -> comfy-capybara, iRule emits HSL events
  - Internal Logging VS (TCP 6514, TLS_Client) -> external_collector_pool -> ASLE
            |
            v
Docker host VMs (Container-Optimized OS):
  - Comfy-capybara: ORAS-pulled compose, five-service stack
  - ASLE: docker-loaded operator-supplied tarball, GCS-polled config bundle
```

UC3 has no F5 Distributed Cloud component. ASLE is configured via a GCS-polling pattern; no operator runner-to-VM channel is required.

## Edits required

### `config/common/gcp/env.json`

```json
{
  "gcp_project_id": "<your-project-id>",
  "gcp_region": "us-west1",
  "gcp_zone": "us-west1-a",
  "project_prefix": "<short-prefix>",
  "resource_owner": "<initials>",
  "admin_src_addr": ["1.2.3.4/32", "10.0.0.0/8"],
  "tf_state_bucket": ""
}
```

- `admin_src_addr` is a JSON array of **quoted CIDR strings** (`IP/prefix`). Bare IPs and unquoted values fail JSON parsing in the workflow.
- Leave `tf_state_bucket` empty; the workflow derives it as `<project_prefix>-state-bucket`.

### `config/uc3/gcp/env.json`

```json
{
  "compute": {
    "gcp_runtime_service_account_email": "<runtime-sa>@<project-id>.iam.gserviceaccount.com",
    "asle_tarball_gcs_uri": "gs://<your-asle-bucket>/asle.tar",
    "asle_image_ref": "<asle-repo>:<tag>",
    "asle_telemetry_port": <port>
  }
}
```

- `asle_tarball_gcs_uri`: where the operator-supplied ASLE image tarball lives.
- `asle_image_ref`: the `repository:tag` `docker load` materializes from the tarball.
- `asle_telemetry_port`: the port ASLE listens on for telemetry ingest. Must be non-zero; the BIG-IP External Collector Pool member uses it.

All other compute / bigip_base / bigip_config / asle_config fields keep their example defaults.

## Deploy

```bash
git checkout -b deploy-adsp-uc3 && git push -u origin deploy-adsp-uc3
```

Watch the workflow in the Actions tab. Modules run: state bucket → infra → (compute || bigip-base) → (bigip-config || asle-config).

`test-adsp-uc3` runs validate only. `destroy-adsp-uc3` tears down in reverse order.

# Part 2: Detailed Reference

## Architecture

```
                            Client
                              │
                          HTTPS 443
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ F5 BIG-IP (Single-NIC, PAYG)                                │
│ ├─ Front-door cert: self-signed, generated by bigip-config  │
│ ├─ Management: 8443 (admin_src_addr only)                   │
│ │                                                            │
│ │     ┌───────── App VIP (443) ─────────────────────────┐  │
│ │     │   pool: comfy_pool (member: comfy-host:8080)     │  │
│ │     │   iRule: HSL → internal_logging_pool             │  │
│ │     └────────────────────────────────────────────────-─┘  │
│ │                                                            │
│ │     ┌──── Internal Logging VS (TCP 6514, TLS_Client) ─┐  │
│ │     │   pool: external_collector_pool                  │  │
│ │     │     (member: asle-host:<asle_telemetry_port>)    │  │
│ │     └───────────────────────────────────────────────--─┘  │
└─────────────────────────────────────────────────────────────┘
              │                                  │
       proxy traffic                       TLS telemetry
              │                                  │
              ▼                                  ▼
┌──────────────────────────┐       ┌────────────────────────────┐
│ Comfy Capybara host      │       │ ASLE host                  │
│ (Docker, COS image)      │       │ (Docker, COS image)        │
│ Pulls compose OCI        │       │ Loads operator-supplied    │
│ artifact at boot via     │       │ image tarball from GCS at  │
│ ORAS; runs five-service  │       │ boot, runs ASLE container  │
│ stack                    │       │                            │
│                          │       │ Poller watches GCS bundle  │
│                          │       │ and POSTs config to ASLE   │
└──────────────────────────┘       └────────────────────────────┘

Subnets: mgmt (/24), ext (/18), int (/18), app (/18)
```

## Deployment flow

```
1. Bootstrap state bucket (GCS) - bucket pre-created out of band
   ↓
2. Terraform: Infra (VPC, subnets, firewall, NAT)
   ↓
   ├──> 3a. Terraform: Compute
   │       (parallel with bigip-base)
   │
   └──> 3b. Terraform: BIG-IP Base
           (parallel with compute)
   ↓
   ├──> 4a. Terraform: BIG-IP Config   (needs compute + bigip-base)
   │
   └──> 4b. Terraform: ASLE Config     (needs bigip-base; parallel with bigip-config)
```

The bigip-config module generates a self-signed RSA cert at apply time and embeds the PEMs (base64) in an AS3 declaration template alongside the rendered iRule body. The declaration is uploaded to `gs://<bucket>/artifacts/uc3/as3/uc3-declaration.json`. BIG-IP polls GCS during its onboarding step and applies the configuration locally. bigip-base accepts the AS3 GCS URI as a direct tfvar, so it does not need to read bigip-config's remote state and can run in parallel with compute.

The asle-config module renders an operations bundle (currently a single `POST /api/bigip` registering the BIG-IP with ASLE) populated from BIG-IP base remote state. The bundle is uploaded to `gs://<bucket>/artifacts/uc3/asle/bundle.json`. A poller container on the ASLE VM watches the bundle's GCS object generation and POSTs each operation to `http://localhost:8000` against the ASLE container. ASLE itself never needs to be reachable from outside the Docker host; all configuration flows through GCS.

Destroy operations run in reverse with parallelism: `(BIG-IP Config || ASLE Config) → (BIG-IP Base || Compute) → Infra`.

## Prerequisites

### GCP

1. GCP project with billing enabled.
2. APIs enabled: Compute Engine, Cloud Resource Manager, Cloud Storage, IAM Service Account Credentials.
3. Pre-create the Terraform state bucket. UC3's deploy workflow names it `<project_prefix>-state-bucket` and expects it to already exist with versioning and public-access-prevention enabled:
   ```bash
   gcloud storage buckets create gs://<project_prefix>-state-bucket \
     --project=<project-id> \
     --location=<region> \
     --uniform-bucket-level-access
   gcloud storage buckets update gs://<project_prefix>-state-bucket \
     --versioning --public-access-prevention
   ```
4. A separate **ASLE tarball bucket** holds the operator-supplied ASLE image tarball. This can be the state bucket or a dedicated bucket; either way the runtime SA needs `roles/storage.objectViewer` on it.
5. Two service accounts:
   - **Deploy SA** - assumed by GitHub Actions via Workload Identity Federation.
   - **Runtime SA** - attached to BIG-IP and to the comfy + ASLE Docker host VMs.

#### Deploy SA roles

| Role | Scope | Why |
|------|-------|-----|
| `roles/compute.networkAdmin` | project | VPC, subnets, router, NAT (infra module). |
| `roles/compute.securityAdmin` | project | Firewall rules (infra module). |
| `roles/compute.instanceAdmin.v1` | project | BIG-IP VM + Docker host VMs (bigip-base, compute modules). |
| `roles/storage.admin` | **bucket** (`<project_prefix>-state-bucket`) | Read/write state objects, the AS3 declaration artifact, and the ASLE config bundle under that bucket. Bind on the bucket only, not project-wide. |
| `roles/iam.serviceAccountUser` | **runtime SA only** | Lets the deploy SA attach the runtime SA to the BIG-IP and Docker host VMs. Do not grant this project-wide. |
| `roles/iam.workloadIdentityUser` | deploy SA itself (via principalSet) | WIF binding so the GitHub OIDC token can assume the deploy SA. |

#### Runtime SA roles (attached to BIG-IP, comfy host, ASLE host)

| Role | Scope | Why |
|------|-------|-----|
| `roles/logging.logWriter` | project | VM telemetry to Cloud Logging. |
| `roles/monitoring.metricWriter` | project | VM telemetry to Cloud Monitoring. |
| `roles/storage.objectViewer` | **bucket** (`<project_prefix>-state-bucket`) | BIG-IP polls the AS3 declaration; ASLE poller watches `bundle.json`. Both under the state bucket. |
| `roles/storage.objectViewer` | **bucket** (`<asle-tarball-bucket>`) | ASLE host's boot script pulls the image tarball with `gsutil cp`. Skip if the tarball lives in the state bucket. |

#### Binding commands

```bash
PROJECT=<project-id>
PREFIX=<project_prefix>
BUCKET=$PREFIX-state-bucket
ASLE_BUCKET=<asle-tarball-bucket>  # may equal $BUCKET
DEPLOY_SA=<deploy-sa>@$PROJECT.iam.gserviceaccount.com
RUNTIME_SA=<runtime-sa>@$PROJECT.iam.gserviceaccount.com

# Deploy SA - project-scoped
for ROLE in roles/compute.networkAdmin roles/compute.securityAdmin \
            roles/compute.instanceAdmin.v1; do
  gcloud projects add-iam-policy-binding "$PROJECT" \
    --member="serviceAccount:$DEPLOY_SA" --role="$ROLE"
done

# Deploy SA - bucket-scoped storage admin on state bucket
gcloud storage buckets add-iam-policy-binding gs://$BUCKET \
  --member="serviceAccount:$DEPLOY_SA" --role="roles/storage.admin"

# Deploy SA - allowed to act-as the runtime SA only
gcloud iam service-accounts add-iam-policy-binding "$RUNTIME_SA" \
  --member="serviceAccount:$DEPLOY_SA" --role="roles/iam.serviceAccountUser"

# Runtime SA - VM telemetry
for ROLE in roles/logging.logWriter roles/monitoring.metricWriter; do
  gcloud projects add-iam-policy-binding "$PROJECT" \
    --member="serviceAccount:$RUNTIME_SA" --role="$ROLE"
done

# Runtime SA - read state-bucket artifacts (AS3 declaration + ASLE bundle)
gcloud storage buckets add-iam-policy-binding gs://$BUCKET \
  --member="serviceAccount:$RUNTIME_SA" --role="roles/storage.objectViewer"

# Runtime SA - read ASLE tarball (skip if tarball lives in the state bucket)
if [ "$ASLE_BUCKET" != "$BUCKET" ]; then
  gcloud storage buckets add-iam-policy-binding gs://$ASLE_BUCKET \
    --member="serviceAccount:$RUNTIME_SA" --role="roles/storage.objectViewer"
fi
```

6. Workload Identity Pool + Provider for GitHub Actions (see [GitHub Secrets Setup](#github-secrets-setup)).
7. Regional quotas: 12 vCPU (BIG-IP `n2-highmem-4` = 4 vCPU; comfy `e2-medium` and ASLE `e2-custom-2-4096` = 4 vCPU; headroom), 4 external IPs (BIG-IP, NAT, optional public IPs on Docker hosts), 150 GB persistent disk SSD.

### F5 BIG-IP

- F5 BIG-IP PAYG image in `projects/f5-7626-networks-public/global/images/`. An example tag ships in `config/uc3/gcp/env.example.json`.
- The image's PAYG license must cover ASM provisioning at `nominal`.

### F5 API Security Local Edition (ASLE)

The ASLE image is operator-supplied and not redistributable.

1. Build or obtain the ASLE Docker image locally and `docker save` it to a tarball.
2. Upload the tarball to a GCS bucket you own.
3. Record the loaded image reference (the `repository:tag` `docker load` will materialize).
4. Decide what port ASLE listens on for telemetry ingest.

Both the tarball URI and the loaded image reference go into `config/uc3/gcp/env.json` under the `compute` block.

### GitHub repository

- Repository forked with Actions enabled.
- Branches:
  - `deploy-adsp-uc3` - validate + plan + apply
  - `test-adsp-uc3` - validate only
  - `destroy-adsp-uc3` - destroy workflow

### Workload Identity Federation

```bash
PROJECT=<project-id>
POOL=github-actions-pool
PROVIDER=github-provider
DEPLOY_SA=<deploy-sa>@$PROJECT.iam.gserviceaccount.com
REPO=<github-org>/<repo>

gcloud iam workload-identity-pools create "$POOL" \
  --project="$PROJECT" --location=global \
  --display-name="GitHub Actions Pool"

gcloud iam workload-identity-pools providers create-oidc "$PROVIDER" \
  --project="$PROJECT" --location=global \
  --workload-identity-pool="$POOL" \
  --display-name="GitHub Provider" \
  --attribute-mapping="google.subject=assertion.sub,attribute.actor=assertion.actor,attribute.repository=assertion.repository" \
  --issuer-uri="https://token.actions.githubusercontent.com"

PROJECT_NUM=$(gcloud projects describe "$PROJECT" --format='value(projectNumber)')
gcloud iam service-accounts add-iam-policy-binding "$DEPLOY_SA" \
  --project="$PROJECT" \
  --role="roles/iam.workloadIdentityUser" \
  --member="principalSet://iam.googleapis.com/projects/$PROJECT_NUM/locations/global/workloadIdentityPools/$POOL/attribute.repository/$REPO"
```

`GCP_WORKLOAD_IDENTITY_PROVIDER` secret value:
`projects/$PROJECT_NUM/locations/global/workloadIdentityPools/$POOL/providers/$PROVIDER`

## Configuration

Two files drive UC3. Copy each `env.example.json` to `env.json` and edit only what's called out.

### Common (`config/common/gcp/env.json`)

```json
{
  "gcp_project_id": "<your-project-id>",
  "gcp_region": "us-west1",
  "gcp_zone": "us-west1-a",
  "project_prefix": "<short-prefix>",
  "resource_owner": "<initials>",
  "admin_src_addr": ["1.2.3.4/32", "10.0.0.0/8"],
  "tf_state_bucket": ""
}
```

- `admin_src_addr`: JSON array of quoted CIDR strings. Each entry must include the prefix (`/32` for a single host). Bare IPs and unquoted values fail JSON parsing in the infra job.
- `tf_state_bucket`: leave empty. The workflow derives it as `<project_prefix>-state-bucket`.

### UC3 GCP / compute / BIG-IP / ASLE (`config/uc3/gcp/env.json`)

The example file pre-enables `features.bigip` and sets:

- **compute**: `e2-medium` Docker hosts on Container-Optimized OS, 20 GB disks, `assign_public_ip: false`. `vm_create_comfy_capybara: true`, `vm_create_asle: true`. Pre-set `comfy_compose_artifact` and `comfy_compose_tag` point at the comfy-capybara compose OCI artifact.
- **bigip_base**: PAYG `n2-highmem-4` image pinned to a known-good tag, single-NIC, 120 GB pd-ssd, ASM at `nominal`.
- **bigip_config**: `backend_compute: "true"`, `awaf_config_payload: "../config/uc3-config.json.tftpl"`. The AS3 template renders with the self-signed cert (generated at apply time), the iRule body, the comfy IP, and the ASLE telemetry endpoint.

The fields you must set are:

```json
{
  "compute": {
    "gcp_runtime_service_account_email": "<runtime-sa>@<project-id>.iam.gserviceaccount.com",
    "asle_tarball_gcs_uri": "gs://<your-asle-bucket>/asle.tar",
    "asle_image_ref": "<asle-repo>:<tag>",
    "asle_telemetry_port": <port>
  }
}
```

- `gcp_runtime_service_account_email`: attached to the BIG-IP and to both Docker host VMs.
- `asle_tarball_gcs_uri`: fully qualified gs:// URI of your uploaded ASLE image tarball.
- `asle_image_ref`: the `repository:tag` `docker load` materializes from your tarball.
- `asle_telemetry_port`: ASLE's telemetry intake port. Must be non-zero; the BIG-IP External Collector Pool member targets this port.

Optional ASLE onboarding overrides go in an `asle_config` block:

```json
{
  "asle_config": {
    "bigip_onboard_name": "uc3-bigip",
    "bigip_onboard_port": "443",
    "bigip_onboard_user": "admin"
  }
}
```

Do not change:

- `bigip_base.nic_count: "false"` - single-NIC architecture.
- `features.bigip: true` - required.

### BIG-IP front-door cert

`f5/bigip-config/gcp/cert.tf` generates a self-signed RSA cert at every apply and embeds it in the AS3 declaration. A re-apply rotates the cert; the BIG-IP picks up the change on its next AS3 poll.

To use a real cert, swap `cert.tf` for a `tls_locally_signed_cert` against your own CA, or pre-provision a Certificate object on the BIG-IP via a separate AS3 declaration and reference it from `uc3-config.json.tftpl`.

### ASLE onboarding bundle

`f5/asle-config/gcp/locals.tf` renders an operations bundle containing a single `POST /api/bigip`. The poller on the ASLE VM watches the bundle's GCS generation and POSTs each operation to `http://localhost:8000`. Adding more operations (OAS uploads, etc.) requires extending the bundle renderer in that file.

## Deployment Procedures

### Initial deploy

1. Fork the repo.
2. Build and upload the ASLE image tarball to a GCS bucket you own; record the gs:// URI and the loaded image reference.
3. Edit the two config files in `config/`.
4. Set GitHub Secrets.
5. Push your edits to your default branch.
6. Cut and push the deploy branch:
   ```bash
   git checkout -b deploy-adsp-uc3 && git push -u origin deploy-adsp-uc3
   ```
7. Watch the workflow in the Actions tab.

### Validation only

```bash
git checkout -b test-adsp-uc3 && git push -u origin test-adsp-uc3
```

Runs `terraform validate` for every module; skips plan and apply.

### Update an existing deployment

1. Edit config on the default branch.
2. Merge or push to `deploy-adsp-uc3`. The workflow re-runs plan + apply.

### Destroy

```bash
git checkout -b destroy-adsp-uc3 && git push -u origin destroy-adsp-uc3
```

Destroy order: (BIG-IP Config || ASLE Config) → (BIG-IP Base || Compute) → Infra. The state bucket is not deleted by the workflow since it was created out of band; remove it manually when you're done. The ASLE tarball is left in place.

```bash
# Through the BIG-IP front door (self-signed cert; -k expected)
curl -kI "https://$MGMT_IP"
```

### AS3 + iRule on the BIG-IP

```bash
ssh admin@$MGMT_IP

tmsh show ltm virtual

tmsh list ltm virtual /Tenant_1/App_1/serviceMain rules

curl -sku admin:"$ADMIN_PASSWORD" \
```

### Comfy capybara on its Docker host

The host is internal-only by default; reach it via the BIG-IP shell:

```bash
COMFY_IP=$(gsutil cat gs://$BUCKET/state/uc3/compute/default.tfstate \
  | jq -r '.outputs.comfy_capybara_internal_ip.value')

# From the BIG-IP, the frontend exposes 8080
curl http://$COMFY_IP:8080
```

### ASLE telemetry intake

The BIG-IP iRule emits one HSL line per HTTP request and one per response, routed through the internal logging VS (TLS) and into ASLE. After traffic hits the app, the ASLE container should reflect the events. Tail the BIG-IP HSL logs as a sanity check:

```bash
# From the BIG-IP
tail -f /var/log/ltm | grep HSL
```

### Operator access to the ASLE management UI

Open item. The default `config/uc3/gcp/env.example.json` sets `assign_public_ip: false` for the Docker hosts, and ASLE config delivery uses GCS polling, so port 8000 on ASLE is only reachable from inside the host.

If you need operator UI access to ASLE:

1. Flip `assign_public_ip: true` in the compute block. This currently affects both Docker hosts (the compute module's `assign_public_ip` is a single module-level var, not per-VM).
2. Open `tcp:8000` from `admin_src_addr` to the management network tag in the infra module (no rule exists today).
3. Reach ASLE at `http://<asle_public_ip>:8000`.

Alternative: jump from the BIG-IP (which has a public IP) to the ASLE internal IP via SSH-tunneled curl.

## Troubleshooting

**`jq: parse error: Invalid numeric literal` in the infra job.**
Your `admin_src_addr` (or another CIDR/list field) has unquoted values. JSON requires `["1.2.3.4/32"]`, not `[1.2.3.4/32]`.

**Bootstrap job fails with `Required "storage.buckets.create" permission` or similar storage error.**
The deploy SA's storage role is bucket-scoped, so the bucket must exist before the workflow runs. Pre-create it with the gcloud commands in [Prerequisites → GCP](#gcp).

**`The user does not have access to service account "<runtime-sa>"` during compute or bigip-base apply.**
The deploy SA needs `roles/iam.serviceAccountUser` on the runtime SA email set in `config/uc3/gcp/env.json` (`compute.gcp_runtime_service_account_email`). Bind on the runtime SA only, not project-wide.

**BIG-IP boots but the AS3 declaration is not applied.**
- Confirm the AS3 declaration is in GCS: `gsutil ls gs://$BUCKET/artifacts/uc3/as3/`.
- SSH to the BIG-IP and check `/var/log/cloud/as3-pull-apply.log`.
- The onboarding template retries `pull_and_apply_as3_from_gcs`; transient failures should self-heal.
- The runtime SA attached to the BIG-IP needs `roles/storage.objectViewer` on the state bucket. See [Runtime SA roles](#runtime-sa-roles-attached-to-big-ip-comfy-host-asle-host).

**Pool members show offline.**
- `gcloud compute instances list` to confirm the Docker hosts are up.
- Internal firewall rules must allow the BIG-IP subnet to reach the app subnet.
- `tmsh show ltm pool members detail` on the BIG-IP.

**Comfy `oras pull` fails at startup.**
- Confirm `comfy_compose_tag` exists in the published OCI artifact at https://github.com/knowbase/comfy-capybara/pkgs/container/comfy-capybara-compose.
- Serial console:
  ```bash
  gcloud compute instances get-serial-port-output <comfy-vm> --project=$PROJECT --zone=$ZONE
  ```

**Comfy compose `up` reports image pull failures.**
The compose pulls images from `ghcr.io/knowbase/comfy-capybara-<service>`. If your fork's images are private, configure docker login on the host before the compose comes up. Confirm `COMFY_TAG` inside the systemd unit matches a published image tag.

**ASLE `gsutil cp` fails on the tarball.**
- Confirm `asle_tarball_gcs_uri` is a fully qualified gs:// URI.
- The runtime SA needs `roles/storage.objectViewer` on the bucket holding the tarball. If that bucket is not the state bucket, bind it explicitly. See [Runtime SA roles](#runtime-sa-roles-attached-to-big-ip-comfy-host-asle-host).
- Test the URI from a workstation with the same SA impersonated.

**`docker load` succeeds but `docker run` fails with `Unable to find image`.**
`asle_image_ref` must exactly match what your tarball materializes. Run `docker load` locally to confirm the loaded reference. Retag and re-save if needed.

**`asle-poller.service` keeps restarting.**
- Journal: `sudo journalctl -u asle-poller.service --no-pager -n 100`.
- The container apt-installs jq and curl at startup; transient network issues during install cause early failures.
- Confirm the bundle is in GCS: `gsutil ls gs://$BUCKET/artifacts/uc3/asle/bundle.json`.
- The poller does not start when `asle_config_bundle_gcs_uri` is empty in the compute tfvars.

**Bundle is present but ASLE never sees the BIG-IP.**
- Poller logs the failing operation to stderr but proceeds.
- Sanity check from inside the ASLE host:
  ```bash
  curl http://localhost:8000/api/bigip
  ```
- If your ASLE revision's API surface differs from `POST /api/bigip`, update `f5/asle-config/gcp/locals.tf` to match.

## Operations

- Restrict `admin_src_addr` to known IP ranges and protect the `deploy-adsp-uc3` / `destroy-adsp-uc3` branches with branch protection rules.
- The BIG-IP front-door cert is self-signed and rotated on every bigip-config apply.
- The ASLE poller is idempotent at the GCS-generation level: a re-applied bundle with identical operations is detected as unchanged. To force a re-onboard, edit `f5/asle-config/gcp/locals.tf` (for example change `bigip_onboard_name`) and re-apply.
- State files should not be edited by hand. Back up state before major changes:
  ```bash
  gsutil -m cp -r gs://$BUCKET/state/uc3 gs://backup-bucket/state-uc3-$(date +%Y%m%d)
  ```
- Destroy environments when idle. BIG-IP PAYG dominates the hourly cost.
