# Deploy Use-Case 3 in Google Cloud

This document provides complete instructions for deploying Application Security and Delivery Portfolio (ADSP) Use-Case 3 in Google Cloud Platform.

---

## Overview

This repository deploys a self-hosted API security demonstration consisting of:

- **Network Infrastructure** - VPC with segmented subnets (management, external, internal, application)
- **Vulnerable Application** - Comfy Capybara running on Docker (replaces UC1's Juice Shop / crAPI workloads)
- **F5 BIG-IP** - Application Delivery Controller with a self-signed front-door TLS cert (single-NIC configuration). Management on 8443 locked to admin source addresses
- **F5 API Security Local Edition (ASLE)** - Self-hosted API security container running on a dedicated Docker host. BIG-IP ships telemetry to it via an iRule and a chain of internal logging pool, internal logging VS (TLS), and external collector pool

UC3 has no F5 Distributed Cloud component. ASLE is configured via a GCS-polling pattern; no operator runner-to-VM channel is required.

The deployment is orchestrated entirely through GitHub Actions using Terraform with GCS remote state. Local execution is not supported.

### Architecture

```
                            Client
                              │
                          HTTPS 443
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ F5 BIG-IP (Single-NIC, PAYG)                                │
│ ├─ Front-door cert: self-signed, generated by bigip-config  │
│ ├─ Management: 8443 (admin_src_addr only)                   │
│ │                                                            │
│ │     ┌───────── App VIP (443) ─────────────────────────┐  │
│ │     │   pool: comfy_pool (member: comfy-host:8080)     │  │
│ │     │   iRule: HSL → internal_logging_pool             │  │
│ │     └────────────────────────────────────────────────-─┘  │
│ │                                                            │
│ │     ┌──── Internal Logging VS (TCP 6514, TLS_Client) ─┐  │
│ │     │   pool: external_collector_pool                  │  │
│ │     │     (member: asle-host:<asle_telemetry_port>)    │  │
│ │     └───────────────────────────────────────────────--─┘  │
└─────────────────────────────────────────────────────────────┘
              │                                  │
       proxy traffic                       TLS telemetry
              │                                  │
              ▼                                  ▼
┌──────────────────────────┐       ┌────────────────────────────┐
│ Comfy Capybara host      │       │ ASLE host                  │
│ (Docker, COS image)      │       │ (Docker, COS image)        │
│ Pulls compose OCI        │       │ Loads operator-supplied    │
│ artifact at boot via     │       │ image tarball from GCS at  │
│ ORAS; runs five-service  │       │ boot, runs ASLE container  │
│ stack                    │       │                            │
│                          │       │ Poller watches GCS bundle  │
│                          │       │ and POSTs config to ASLE   │
└──────────────────────────┘       └────────────────────────────┘

Subnets: mgmt (/24), ext (/18), int (/18), app (/18)
```

### Deployment Flow

The GitHub Actions workflow deploys modules with the following dependencies:

```
1. Bootstrap State Bucket (GCS)
   ↓
2. Terraform: Infra (VPC, subnets, firewall, NAT)
   ↓
   ├──> 3a. Terraform: Compute
   │       (parallel with bigip-base)
   │
   └──> 3b. Terraform: BIG-IP Base
           (parallel with compute)
   ↓
   ├──> 4a. Terraform: BIG-IP Config   (needs compute + bigip-base)
   │
   └──> 4b. Terraform: ASLE Config     (needs bigip-base; parallel with bigip-config)
```

**AS3 Configuration Flow:**
- The bigip-config module generates a self-signed RSA cert at apply time and embeds the PEMs (base64) in an AS3 declaration template alongside the rendered iRule body
- The declaration is uploaded to GCS at `gs://{state-bucket}/artifacts/uc3/as3/uc3-declaration.json`
- BIG-IP polls GCS during its `pull_and_apply_as3_from_gcs` onboarding step and applies the configuration locally
- bigip-base accepts the AS3 GCS URI as a direct tfvar, so it does not need to read bigip-config's remote state and can run in parallel with compute

**ASLE Configuration Flow:**
- The asle-config module renders an operations bundle (currently a single `POST /api/bigip` registering the BIG-IP with ASLE) populated from the BIG-IP base remote state outputs
- The bundle is uploaded to `gs://{state-bucket}/artifacts/uc3/asle/bundle.json`
- A poller container running on the ASLE VM (started at boot, long-running) watches the bundle's GCS object generation, downloads on change, and POSTs each operation to `http://localhost:8000` against the ASLE container
- The poller is enabled only when `asle_config_bundle_gcs_uri` is non-empty
- ASLE itself never needs to be reachable from outside the Docker host; all configuration flows through GCS

Destroy operations run in reverse with parallelism: `(BIG-IP Config || ASLE Config) → (BIG-IP Base || Compute) → Infra → State Bucket`

## Prerequisites

### GCP Requirements

1. **GCP Project** with billing enabled
2. **Required APIs enabled:**
   - Compute Engine API
   - Cloud Resource Manager API
   - Cloud Storage API
   - IAM Service Account Credentials API
3. **Service Account** with the following roles:
   - `roles/compute.admin`
   - `roles/storage.admin`
   - `roles/iam.serviceAccountUser`
4. **Workload Identity Pool** configured for GitHub Actions federation
5. **Sufficient Quotas:**
   - CPUs: 12+ (BIG-IP `n2-highmem-4` = 4 vCPU, comfy + ASLE Docker hosts = 4 vCPU, headroom)
   - External IP addresses: 4+ (BIG-IP, NAT, optional public IPs on Docker hosts)
   - Persistent disk SSD: 150+ GB
6. **ASLE tarball bucket** - the operator-supplied ASLE image tarball lives in a GCS bucket the runtime SA can read. Can be the state bucket or a dedicated bucket; bind `roles/storage.objectViewer` on whichever one the runtime SA needs to reach.

#### Hardening: tighter deploy SA IAM (optional)

The roles above are the simplest set that lets the workflow run end-to-end. For least-privilege, replace `roles/compute.admin` with the narrower split:

- `roles/compute.networkAdmin` (VPC, subnets, router, NAT)
- `roles/compute.securityAdmin` (firewall rules)
- `roles/compute.instanceAdmin.v1` (BIG-IP + Docker host VMs)

`roles/storage.admin` can be swapped for a custom storage-admin role with `storage.buckets.{create,get,update}` + `storage.objects.{create,get,delete,list}`. Bind `roles/iam.serviceAccountUser` only on the runtime SA attached to VMs, not project-wide.

#### Runtime SA (attached to BIG-IP and the Docker host VMs)

The runtime SA referenced by `compute.gcp_runtime_service_account_email` in `config/uc3/gcp/env.json` carries:

- `roles/logging.logWriter`
- `roles/monitoring.metricWriter`
- `roles/storage.objectViewer` on the state bucket (BIG-IP polls AS3; ASLE poller watches `bundle.json`)
- `roles/storage.objectViewer` on the ASLE tarball bucket (skip if that's the state bucket)

### F5 BIG-IP Requirements

1. **F5 BIG-IP PAYG image** in `projects/f5-7626-networks-public/global/images/` (an example tag is in `config/uc3/gcp/env.example.json`)
2. The image's PAYG license covers ASM provisioning at `nominal`

### F5 API Security Local Edition (ASLE) Requirements

The ASLE image is operator-supplied and not redistributable. To deploy:

1. **Build or obtain** the ASLE Docker image locally and `docker save` it to a tarball
2. **Upload the tarball** to a GCS bucket you own. The runtime SA must have `roles/storage.objectViewer` on that bucket
3. **Record** the loaded image reference (the `repository:tag` `docker load` materializes from your tarball)
4. **Decide** what port ASLE listens on for telemetry ingest (the value the BIG-IP External Collector Pool will target)

Both the tarball URI and the loaded image reference go into `config/uc3/gcp/env.json` under the `compute` block.

### GitHub Repository Requirements

1. **Forked Repository** with Actions enabled
2. **Protected Branches:**
   - `deploy-adsp-uc3` - triggers validation + deployment
   - `test-adsp-uc3` - triggers validation only
   - `destroy-adsp-uc3` - triggers destroy workflow

### GitHub Secrets Setup

Configure the following secrets in GitHub repository settings: `Settings → Secrets and variables → Actions → New repository secret`

#### Required Secrets

| Secret Name | Description | How to Obtain |
|-------------|-------------|---------------|
| `GCP_WORKLOAD_IDENTITY_PROVIDER` | Workload Identity Provider resource name | Format: `projects/PROJECT_NUMBER/locations/global/workloadIdentityPools/POOL_ID/providers/PROVIDER_ID` |
| `GCP_SERVICE_ACCOUNT` | Deploy service account email | Format: `SERVICE_ACCOUNT_NAME@PROJECT_ID.iam.gserviceaccount.com` |

## Quickstart

The minimum edits to get UC3 running, assuming you already have GCP + WIF + F5 entitlements wired up per the Prerequisites section above.

### `config/common/gcp/env.json`

```json
{
  "gcp_project_id": "<your-project-id>",
  "gcp_region": "us-west1",
  "gcp_zone": "us-west1-a",
  "project_prefix": "<short-prefix>",
  "resource_owner": "<initials>",
  "admin_src_addr": ["1.2.3.4/32", "10.0.0.0/8"],
  "tf_state_bucket": ""
}
```

- `admin_src_addr` is a JSON array of **quoted CIDR strings** (`IP/prefix`). Bare IPs and unquoted values fail JSON parsing in the workflow.
- Leave `tf_state_bucket` empty; the workflow derives it as `<project_prefix>-state-bucket`.

### `config/uc3/gcp/env.json`

```json
{
  "compute": {
    "gcp_runtime_service_account_email": "<runtime-sa>@<project-id>.iam.gserviceaccount.com",
    "asle_tarball_gcs_uri": "gs://<your-asle-bucket>/asle.tar",
    "asle_image_ref": "<asle-repo>:<tag>",
    "asle_telemetry_port": <port>
  }
}
```

- `gcp_runtime_service_account_email`: attached to BIG-IP and to both Docker host VMs
- `asle_tarball_gcs_uri`: where the operator-supplied ASLE image tarball lives
- `asle_image_ref`: the `repository:tag` `docker load` materializes from the tarball
- `asle_telemetry_port`: the port ASLE listens on for telemetry ingest. Must be non-zero; the BIG-IP External Collector Pool member uses it

Every other compute / bigip_base / bigip_config / asle_config field keeps its example default.

### Deploy

```bash
git checkout -b deploy-adsp-uc3 && git push -u origin deploy-adsp-uc3
```

Watch the workflow in the Actions tab. Modules run: state bucket → infra → (compute || bigip-base) → (bigip-config || asle-config).

`test-adsp-uc3` runs validate only. `destroy-adsp-uc3` tears down in reverse order.

## Configuration

### Step 1: Configure Common Settings

Edit `config/common/gcp/env.json`:

```json
{
  "gcp_project_id": "your-gcp-project-id",
  "gcp_region": "us-west1",
  "gcp_zone": "us-west1-a",
  "project_prefix": "your-prefix",
  "resource_owner": "your-initials",
  "admin_src_addr": ["1.2.3.4/32", "10.0.0.0/8"],
  "tf_state_bucket": ""
}
```

**Required Changes:**
- `gcp_project_id` - Your GCP project ID
- `gcp_region` - Target GCP region
- `gcp_zone` - Target GCP zone (must be inside `gcp_region`)
- `project_prefix` - Unique prefix for resource naming (lowercase, alphanumeric)
- `resource_owner` - Your initials or identifier for resource tagging
- `admin_src_addr` - Public IP CIDRs allowed to reach management interfaces. Each entry is a quoted CIDR (`/32` for a single host); bare IPs and unquoted values fail JSON parsing.

**Leave as-is:**
- `tf_state_bucket` - auto-generated as `${project_prefix}-state-bucket`

### Step 2: Configure Use-Case Settings

Edit `config/uc3/gcp/env.json`:

```json
{
  "compute": {
    "gcp_runtime_service_account_email": "<runtime-sa>@<project-id>.iam.gserviceaccount.com",
    "machine_type": "e2-medium",
    "boot_disk_gb": 20,
    "assign_public_ip": false,
    "enable_oslogin": false,
    "install_docker_compose": true,
    "vm_create_juice_shop": false,
    "vm_create_crapi": false,
    "vm_create_comfy_capybara": true,
    "vm_create_asle": true,
    "comfy_compose_artifact": "ghcr.io/knowbase/comfy-capybara-compose",
    "comfy_compose_tag": "0.3.0",
    "asle_tarball_gcs_uri": "gs://your-bucket/path/to/asle.tar",
    "asle_image_ref": "your-asle-repo:tag",
    "asle_management_port": 8000,
    "asle_telemetry_port": 0
  },
  "bigip_base": {
    "image_name": "projects/f5-7626-networks-public/global/images/f5-bigip-21-0-0-0-0-10-payg-best-plus-25mbps-251104081529",
    "machine_type": "n2-highmem-4",
    "min_cpu_platform": "Intel Cascade Lake",
    "disk_type": "pd-ssd",
    "disk_size_gb": 120,
    "f5_username": "admin",
    "nic_count": "false",
    "asm": "nominal",
    "apm": "none"
  },
  "bigip_config": {
    "backend_compute": "true",
    "awaf_config_payload": "../config/uc3-config.json.tftpl"
  },
  "features": {
    "bigip": true
  }
}
```

**Required Changes (compute):**
- `gcp_runtime_service_account_email` - Service account attached to BIG-IP and both Docker host VMs
- `asle_tarball_gcs_uri` - gs:// URI of your uploaded ASLE image tarball
- `asle_image_ref` - the `repository:tag` `docker load` materializes from your tarball
- `asle_telemetry_port` - port ASLE listens on for telemetry ingest. Must be non-zero or the BIG-IP External Collector Pool member is unusable

**Customizable Settings:**
- `comfy_compose_tag` - pin a specific Comfy Capybara release; also drives `COMFY_TAG` inside the compose
- `comfy_compose_artifact` - override if pulling from a fork
- `asle_management_port` - ASLE management UI port (default 8000)
- `vm_create_juice_shop` / `vm_create_crapi` - UC1 workloads, left available for backwards compatibility. UC3 uses comfy_capybara.
- `bigip_base.machine_type` - BIG-IP instance size (minimum: n2-highmem-4)
- `bigip_base.asm` - ASM provisioning level: `none`, `nominal`, `minimum`, `dedicated`

**Do Not Modify:**
- `bigip_base.nic_count: "false"` - single-NIC architecture (hardcoded)
- `features.bigip: true` - required for BIG-IP deployment

### Step 3: Configure BIG-IP Defaults

The `bigip_base` and `bigip_config` blocks shown above are sufficient for a standard demo. `awaf_config_payload` is pinned to `../config/uc3-config.json.tftpl`, the UC3 AS3 template. The self-signed front-door cert is generated by Terraform at apply time and embedded in the declaration; no operator action is required.

If a real certificate is needed in place of the self-signed cert, swap `f5/bigip-config/gcp/cert.tf` for a `tls_locally_signed_cert` against your own CA, or pre-provision a Certificate object in BIG-IP via a separate AS3 declaration and reference it from `uc3-config.json.tftpl`.

### Step 4: Configure ASLE Onboarding Bundle

The asle-config module renders the operations bundle from defaults. The current bundle contains a single `POST /api/bigip` registering the BIG-IP with ASLE. Defaults in `config/uc3/gcp/env.json` can override the registered name, port, or user under an `asle_config` block:

```json
{
  "asle_config": {
    "bigip_onboard_name": "uc3-bigip",
    "bigip_onboard_port": "443",
    "bigip_onboard_user": "admin"
  }
}
```

Additional operations (OAS uploads, etc.) require an extension to the bundle renderer in `f5/asle-config/gcp/locals.tf`.

## Deployment Procedures

### Initial Deployment

1. **Fork this repository** to your GitHub organization or account
2. **Build and upload your ASLE image tarball** to a GCS bucket you own. Record the gs:// URI and the loaded image reference.
3. **Configure files:**
   - `config/common/gcp/env.json`
   - `config/uc3/gcp/env.json` (filling in the ASLE fields with the values from step 2)
4. **Set GitHub Secrets** (see GitHub Secrets Setup)
5. **Commit and push changes** to `main` branch
6. **Create deployment branch:**
   ```bash
   git checkout -b deploy-adsp-uc3
   git push origin deploy-adsp-uc3
   ```
7. **Monitor workflow execution** in GitHub Actions tab
8. **Retrieve outputs** (see Accessing Deployment Outputs)

### Validation-Only Testing

```bash
git checkout -b test-adsp-uc3
git push origin test-adsp-uc3
```

This triggers validation for all modules but skips `terraform apply` steps.

### Updating Existing Deployment

Make configuration changes on `main`, then push to `deploy-adsp-uc3`. The workflow plans and applies changes automatically.

### Destroying Infrastructure

**WARNING:** This permanently deletes all resources including the state bucket.

```bash
git checkout -b destroy-adsp-uc3
git push origin destroy-adsp-uc3
```

Destroy sequence:
1. BIG-IP Config and ASLE Config (parallel)
2. BIG-IP Base and Compute (parallel)
3. Infra
4. GCS state bucket (including all history; this removes the AS3 declaration and the asle-config bundle. The ASLE tarball is left in place if it lives in a separate bucket.)

### BIG-IP Management Access

```bash
MGMT_IP=$(gsutil cat gs://${STATE_BUCKET}/state/uc3/bigip-base/default.tfstate | \
  jq -r '.outputs.bigip_public_ip.value')
ADMIN_PASSWORD=$(gsutil cat gs://${STATE_BUCKET}/state/uc3/bigip-base/default.tfstate | \

echo "BIG-IP GUI: https://${MGMT_IP}:8443"
echo "User: admin"
echo "Password: ${ADMIN_PASSWORD}"
```

### Application Access

```bash
# Through the BIG-IP front door (self-signed cert; -k expected)
curl -kI "https://${MGMT_IP}"
```

The front door listens on 443. The browser will warn on the self-signed cert.

### Health Checks

**AS3 + iRule on BIG-IP:**

```bash
ssh admin@${MGMT_IP}

tmsh show ltm virtual
tmsh show ltm pool members

# Confirm the iRule is attached
tmsh list ltm virtual /Tenant_1/App_1/serviceMain rules

# AS3 declaration
curl -sku admin:${ADMIN_PASSWORD} \
  https://localhost:8443/mgmt/shared/appsvcs/declare | jq
```

**Comfy Capybara on its Docker host:**

The host is internal-only by default; the BIG-IP reaches it over the VPC. To check from the BIG-IP shell:

```bash
COMFY_IP=$(gsutil cat gs://${STATE_BUCKET}/state/uc3/compute/default.tfstate | \
  jq -r '.outputs.comfy_capybara_internal_ip.value')

# From BIG-IP, frontend exposes 8080
curl http://${COMFY_IP}:8080
```

**ASLE telemetry intake:**

The BIG-IP iRule emits one HSL line per HTTP request and one per response, routed through the internal logging VS (TLS) and into ASLE. After traffic hits the app, the ASLE container should reflect the events. Tail the BIG-IP HSL logs as a sanity check:

```bash
# From BIG-IP
tail -f /var/log/ltm | grep HSL
```

**Operator Access to ASLE Management UI:**

Open item. The default `config/uc3/gcp/env.example.json` sets `assign_public_ip: false` for the Docker hosts. The ASLE config delivery path uses GCS polling and never needs ASLE to be externally reachable, so the ASLE container's port 8000 is currently only accessible from inside the Docker host.

If operator UI access to ASLE is required:

1. Flip `assign_public_ip: true` in the compute block (this currently affects both Comfy and ASLE hosts; the compute module's `assign_public_ip` is a single module-level var, not per-VM)
2. Open `tcp:8000` from `admin_src_addr` to the management network tag in the infra module (no rule exists today)
3. Reach ASLE at `http://<asle_public_ip>:8000`

Alternative: jump from the BIG-IP (which has a public IP) to the ASLE internal IP via SSH-tunneled curl.

## Troubleshooting

### GCP Authentication Failures

**Error:** `Error: google: could not find default credentials`

**Resolution:**
- Verify `GCP_WORKLOAD_IDENTITY_PROVIDER` secret is correctly formatted
- Verify `GCP_SERVICE_ACCOUNT` matches the service account with WIF binding
- Service account must have `roles/compute.admin` and `roles/storage.admin`

### Service Account `actAs` Errors

**Error:** `The user does not have access to service account "<runtime-sa>"` during compute or bigip-base apply

**Resolution:**
- The deploy SA needs `roles/iam.serviceAccountUser` on the runtime SA email set in `config/uc3/gcp/env.json` (`compute.gcp_runtime_service_account_email`)
- Bind on the runtime SA only, not project-wide

### Remote State Errors

**Error:** `Error: Failed to get existing workspaces: storage: bucket doesn't exist`

**Resolution:**
- State bucket is auto-created by the bootstrap job. Verify it completed
- Check: `gsutil ls -p PROJECT_ID | grep state-bucket`

**Error:** `Error acquiring the state lock`

**Resolution:**
- Another workflow run may be in progress
- If stuck, manually remove the lock:
  ```bash
  gsutil ls gs://${STATE_BUCKET}/state/uc3/MODULE_NAME/default.tflock
  gsutil rm gs://${STATE_BUCKET}/state/uc3/MODULE_NAME/default.tflock
  ```

### BIG-IP Deployment Issues

**Error:** BIG-IP boots but the AS3 declaration is not applied

**Resolution:**
- Check that the AS3 declaration was uploaded: `gsutil ls gs://${STATE_BUCKET}/artifacts/uc3/as3/`
- SSH to the BIG-IP and check `/var/log/cloud/as3-pull-apply.log`
- The onboarding template includes a `pull_and_apply_as3_from_gcs` step that retries; transient failures should self-heal
- The runtime SA attached to the BIG-IP needs `roles/storage.objectViewer` on the state bucket

**Error:** Pool members show offline

**Resolution:**
- Verify the Comfy and ASLE Docker hosts are running: `gcloud compute instances list`
- Check internal firewall rules permit BIG-IP subnet to app subnet
- SSH to the BIG-IP and check `tmsh show ltm pool members detail`

### Comfy Capybara Won't Come Up

**Error:** Comfy startup fails on `oras pull`

**Resolution:**
- Confirm `comfy_compose_tag` exists in the published OCI artifact. Browse https://github.com/knowbase/comfy-capybara/pkgs/container/comfy-capybara-compose for available tags
- Check the Docker host's serial console output:
  ```bash
  gcloud compute instances get-serial-port-output ${COMFY_VM_NAME} \
    --project=PROJECT_ID --zone=ZONE
  ```

**Error:** Comfy compose `up` reports image pull failures

**Resolution:**
- The compose pulls images from `ghcr.io/knowbase/comfy-capybara-<service>`. If the registry is private, set `imagePullSecrets`-equivalent via a docker login on the host before the compose comes up
- Confirm the `COMFY_TAG` env var inside the systemd unit matches a published image tag

### ASLE Won't Come Up

**Error:** `gsutil cp` fails on the tarball

**Resolution:**
- Confirm `asle_tarball_gcs_uri` is a fully qualified gs:// URI
- The runtime SA (`gcp_runtime_service_account_email`) must have `roles/storage.objectViewer` on the bucket holding the tarball
- Test the URI manually from a workstation with the same SA impersonated

**Error:** `docker load` succeeds but `docker run` fails with `Unable to find image`

**Resolution:**
- The image reference in `asle_image_ref` must exactly match what your tarball materializes. Run `docker load` locally to confirm the loaded ref
- If your tarball is built from an image with a different tag, retag before saving

### ASLE Poller

**Error:** `asle-poller.service` keeps restarting

**Resolution:**
- Check journal: `sudo journalctl -u asle-poller.service --no-pager -n 100`
- The container apt-installs jq and curl at startup; transient network issues during install cause early failures
- Confirm the bundle exists: `gsutil ls gs://${STATE_BUCKET}/artifacts/uc3/asle/bundle.json`
- The poller does not start if `asle_config_bundle_gcs_uri` is empty in the compute tfvars

**Error:** Bundle is present but ASLE never sees the BIG-IP

**Resolution:**
- Poller logs the failing operation to stderr but proceeds
- Sanity-check from inside the Docker host:
  ```bash
  curl http://localhost:8000/api/bigip
  ```
- If ASLE's API surface differs from `POST /api/bigip` (the user-supplied ASLE revision may differ), update `f5/asle-config/gcp/locals.tf` to match

### Configuration Errors

**Error:** `jq: parse error: Invalid numeric literal` in the infra job

**Resolution:**
- Your `admin_src_addr` (or another CIDR/list field) has unquoted values
- JSON requires `["1.2.3.4/32"]`, not `[1.2.3.4/32]`

### Quota Exceeded Errors

**Error:** `Quota 'CPUS' exceeded. Limit: X in region Y`

**Resolution:**
- Request quota increase in GCP Console: `IAM & Admin → Quotas`
- Reduce BIG-IP instance size in `config/uc3/gcp/env.json` (minimum: n2-highmem-4)

## Best Practices for Forkers

### Security

- **Never commit secrets** to the repository
- **Use GitHub Secrets** for all sensitive values (passwords, certificates, API keys)
- **Rotate credentials regularly** (service account keys)
- **Restrict `admin_src_addr`** to known IP addresses only
- **Enable branch protection** on `deploy-*` and `destroy-*` branches
- **Review firewall rules** before deployment in production environments

### Configuration Management

- **Keep `env.json` files non-secret** - they should contain no credentials
- **Use meaningful `project_prefix`** values to avoid naming collisions
- **Tag resources** using `resource_owner` for cost tracking
- **Version control all changes** to configuration files
- **Test changes** on `test-adsp-uc3` branch before deploying

### State Management

- **Do not edit state files manually**
- **Enable versioning** on state bucket (auto-enabled by workflow)
- **Back up state** before major changes:
  ```bash
  gsutil -m cp -r gs://${STATE_BUCKET}/state/uc3 gs://backup-bucket/state-uc3-$(date +%Y%m%d)
  ```
- **Clean up old state** after successful destroys
- **The BIG-IP front-door cert is self-signed and rotated on every bigip-config apply.** A re-apply produces a new declaration MD5, which the BIG-IP picks up on its next AS3 poll.
- **The ASLE poller is idempotent at the GCS-generation level.** A re-applied bundle with identical operations is detected as unchanged. To force a re-onboard, edit `f5/asle-config/gcp/locals.tf` (for example change `bigip_onboard_name`) and re-apply.

### Cost Optimization

- **Destroy environments when not in use** (demo/test scenarios)
- **Use minimal instance sizes** for non-production:
  - Docker hosts: `e2-medium` (default) or shrink to `e2-small` for capability demos
  - BIG-IP: `n2-highmem-4` (minimum supported)
- **Monitor costs** using GCP Billing Reports
- **Set billing alerts** to avoid unexpected charges

### Workflow Management

- **Use descriptive commit messages** when triggering deployments
- **Monitor GitHub Actions logs** during deployment
- **Review Terraform plans** before approving apply steps
- **Document customizations** in repository README or wiki
- **Test destroy workflow** in non-production before using in production

### Naming Conventions

- **project_prefix:** Lowercase, alphanumeric, max 10 characters
- **resource_owner:** 2-4 character initials or identifier
- **Branch names:** Follow existing pattern (`deploy-`, `test-`, `destroy-`)
